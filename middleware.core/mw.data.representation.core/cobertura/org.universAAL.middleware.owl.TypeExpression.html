 /**
 31   
  * A class for the concept of OWL class expressions and OWL data ranges, which
 32   
  * represent sets of individuals by formally specifying conditions on the
 33   
  * individuals' properties, or sets of literals by restricting primitive data
 34   
  * types. Example conditions are intersection of individuals, or restrictions.
 35   
  * Each type expression represents a set of individuals (just like a class in an
 36   
  * object-oriented programming language (OOP) represents a set of objects), or a
 37   
  * set of literals.
 38   
  *
 39   
  * <p>
 40   
  * OWL-2 explicitly distinguishes between <i>class expressions</i> (to describe
 41   
  * sets of individuals) and <i>data ranges</i> (to describe sets of literals).
 42   
  * The terminology is as follows:
 43   
  * <ul>
 44   
  * <li>class expression: represents a set of individuals; a special kind of a
 45   
  * class expression is the direct definition of a class (with {@link TypeURI})
 46   
  * which corresponds to a class in OOP.
 47   
  * <li>individual: instance of a class; corresponds to an instance of a class in
 48   
  * OOP.
 49   
  * <li>data range: represents a set of literals; a special kind of a data range
 50   
  * is the data type (with {@link TypeURI}), like int or string.
 51   
  * <li>literal: instance of a data range, e.g. the literal '1' is an instance of
 52   
  * int, and the literal "Hello World" is an instance of String.
 53   
  * </ul>
 54   
  *
 55   
  * <p>
 56   
  * In universAAL, there is only one class for each concept of class expression
 57   
  * and data range. For example, OWL-2 distinguishes between <i>ObjectOneOf</i>
 58   
  * (to enumerate individuals) and <i>DataOneOf</i> (to enumerate literals),
 59   
  * whereas universAAL has only the one class {@link Enumeration}. The drawback
 60   
  * of this simplification is that an RDF-graph should be created starting with
 61   
  * the leafs towards the root of the graph. For example:<br>
 62   
  *
 63   
  * <pre>
 64   
  * // problematic:
 65   
  * u = new Union();
 66   
  * u.addType(new Enumeration(..));
 67   
  *
 68   
  * // OK:
 69   
  * u = new Union(new Enumeration(..));
 70   
  * </pre>
 71   
  *
 72   
  * In this example, the type of the union (whether it is a class expression or a
 73   
  * data range) can be determined because the argument is already fully set up
 74   
  * and it is known whether the given enumeration is a class expression or a data
 75   
  * range.
 76   
  *
 77   
  * <p>
 78   
  * There are three different kinds of type expressions:
 79   
  *
 80   
  * <ul>
 81   
  * <li>{@link PropertyRestriction}: restricts the set of individuals by defining
 82   
  * a restriction on a property of these individuals. Can be used only for class
 83   
  * expressions.<br>
 84   
  * See also: {@link HasValueRestriction}, {@link AllValuesFromRestriction},
 85   
  * {@link SomeValuesFromRestriction}, <br>
 86   
  * {@link MinCardinalityRestriction}, {@link ExactCardinalityRestriction},
 87   
  * {@link MaxCardinalityRestriction}
 88   
  * <li>Set-theoretic expressions: enumeration of individuals and all standard
 89   
  * Boolean connectives. Can be used for class expressions and for data
 90   
  * ranges.<br>
 91   
  * See also: {@link TypeURI}, {@link Enumeration},<br>
 92   
  * {@link Union}, {@link Intersection}, {@link Complement}
 93   
  * <li>{@link TypeRestriction}: restricts the set literals. Can be used only for
 94   
  * data ranges.<br>
 95   
  * See also: {@link BoundedValueRestriction}, {@link LengthRestriction}
 96   
  * </ul>
 97   
  *
 98   
  * <p>
 99   
  * There are three main methods to compare type expression:
 100   
  * <ul>
 101   
  * <li>hasMember: determines if the given object is a member of the class
 102   
  * represented by this type expression</li>
 103   
  * <li>matches: determines if the given type expression is a subset of the class
 104   
  * represented by this type expression</li>
 105   
  * <li>isDisjointWith: determines if the given type expression has no member in
 106   
  * common with the class represented by this type expression</li>
 107   
  * </ul>
 108   
  *
 109   
  * <p>
 110   
  * All three methods have three parameters in common:
 111   
  * <ul>
 112   
  * <li>context: context of the matchmaking. The <code>context</code> table maps
 113   
  * the URIs of certain variables onto values currently assigned to them. The
 114   
  * variables are either standard variables managed by the universAAL middleware
 115   
  * or parameters of a specific service in whose context this method is called.
 116   
  * If there is already a value assigned to such a referenced variable, it must
 117   
  * be replaced by the associated value, otherwise this method may expand the
 118   
  * <code>context</code> table by deriving a value for such unassigned but
 119   
  * referenced variables with which the goal of the method can be asserted. In
 120   
  * case of returning true, the caller must check the size of the
 121   
  * <code>context</code> table to see if new conditions are added in. If the
 122   
  * <code>context</code> map is null, the method does a global and unconditional
 123   
  * check.</li>
 124   
  * <li>ttl: time-to-live parameter. For each subsequent call to one of these
 125   
  * methods the ttl parameter is decreased by one. If it reaches zero, an
 126   
  * {@link IllegalArgumentException} is thrown with the text provided in
 127   
  * {@link #EXCEPTION_TTL}. This can be used to avoid cycles in the type
 128   
  * expression.</li>
 129   
  * <li>log: list that can be filled with log entries and that shows at which
 130   
  * point the matchmaking failed. May be null.</li>
 131   
  * </ul>
 132   
  *
 133   
  * @see <a href=
 134   
  *      "https://github.com/universAAL/middleware/wiki/Data-Representation#Type_expression">
 135   
  *      Data Representation Wiki page</a>
 136
